# AI Agent / RAG 실무 개념 정리  
## Top-K, LLM 사용, 재인덱스 제어 + Chunk / Embedding / Rerank / Vector DB / Tool Calling

---

## 1. 전체 그림부터 보기

AI Agent나 RAG(Retrieval-Augmented Generation) 시스템은 보통 아래 흐름으로 동작합니다.

1. 문서 수집
2. 문서 분할(Chunking)
3. 임베딩 생성(Embedding)
4. 벡터 DB 또는 검색 인덱스에 저장(Indexing)
5. 사용자 질문 입력
6. 질문도 임베딩 또는 검색 질의로 변환
7. Top-K 검색
8. 필요 시 Rerank
9. LLM이 문맥 이해 및 추론
10. 필요 시 Tool/API 호출
11. 최종 응답 생성
12. 문서 변경 시 재인덱스(Reindex)

즉, AI Agent는 단순히 LLM 하나만 있는 것이 아니라:

- **지식 저장소**
- **검색 계층**
- **판단 엔진(LLM)**
- **행동 도구(Tools)**
- **운영 관리(Reindex, versioning, observability)**

가 함께 작동하는 구조입니다.

---

## 2. 핵심 용어 한 줄 정의

### Top-K
검색 결과 중 **상위 K개 문서/청크만 선택하는 값**

### LLM 사용
검색된 정보와 사용자 요청을 바탕으로 **이해, 추론, 요약, 계획, 도구 선택, 응답 생성**을 수행하는 것

### 재인덱스 제어
문서가 바뀌거나 검색 구조가 바뀔 때 **언제 다시 인덱스를 만들지 관리하는 것**

### Chunk
긴 문서를 검색에 적합하도록 **작은 단위로 나눈 조각**

### Embedding
문장/문서를 숫자 벡터로 바꿔 **의미적 유사도 검색**이 가능하게 하는 표현

### Vector DB
임베딩 벡터를 저장하고 **유사도 기반 검색**을 빠르게 수행하는 저장소

### Rerank
Top-K 후보를 한 번 더 정교하게 재정렬해서 **진짜 관련도 높은 순서**로 다시 배치하는 단계

### Tool Calling
LLM이 외부 기능(DB 조회, API 호출, 계산기, 사내 시스템 등)을 사용해 **행동하는 것**

---

## 3. 왜 이런 구성들이 필요한가

기존 검색 시스템은 보통 키워드 중심입니다.

예:
- 사용자가 “연차 이월 기준” 검색
- 문서에 “연차”와 “이월”이 많이 들어간 문서가 나옴

하지만 실제 질문은 표현이 다양합니다.

예:
- “남은 휴가를 내년으로 넘길 수 있나?”
- “연차 소멸되기 전에 이월 규정이 뭐지?”
- “미사용 휴가 carry-over 정책 알려줘”

이런 경우 단순 키워드 검색보다 **의미 기반 검색**이 더 강합니다.
그래서 Embedding + Vector DB + Top-K + Rerank + LLM 조합이 자주 사용됩니다.

---

## 4. Chunk: 왜 문서를 잘게 나누나

긴 문서를 통째로 넣으면 문제가 많습니다.

### 문제점
- 검색 정확도가 떨어짐
- 관련 없는 내용이 같이 섞임
- LLM 입력 토큰이 낭비됨
- 정확한 근거 위치를 찾기 어려움

그래서 문서를 적절한 크기로 자릅니다. 이것이 Chunk입니다.

### 예시
원문:
- 사내 규정집 200페이지 PDF

Chunking 후:
- 1장 채용 규정
- 2장 휴가 규정
- 3장 연차 이월 규정
- 4장 경조사 휴가 규정
- ...

또는 더 잘게:
- 400~800 tokens 단위
- 문단 단위
- 제목 + 본문 단위
- 표/섹션 단위

### Chunk 설계 포인트
- 너무 작으면 문맥이 끊김
- 너무 크면 노이즈가 많아짐
- overlap(겹침)을 두면 문맥 연결이 좋아짐

예:
- chunk size: 500 tokens
- overlap: 80 tokens

실무에서는 정답이 하나가 아니라, 데이터 특성에 따라 다르게 조정합니다.

---

## 5. Embedding: 의미를 숫자로 바꾸는 것

Embedding은 텍스트를 벡터로 변환하는 기술입니다.

예:
- “환불 정책”
- “반품 규정”
- “refund policy”

이 세 표현은 단어는 달라도 의미가 비슷하므로, 벡터 공간에서 가까운 위치에 배치되도록 학습됩니다.

### 왜 중요한가
사용자가 문서에 없는 표현으로 질문해도,
비슷한 의미의 문서를 찾을 수 있게 해줍니다.

### 쉽게 비유하면
문장을 “뜻 좌표”로 바꾸는 것입니다.

- 비슷한 뜻 = 가까운 좌표
- 다른 뜻 = 먼 좌표

### 실무 포인트
임베딩 모델이 바뀌면:
- 기존 벡터와 호환이 안 될 수 있음
- 전체 재인덱스가 필요할 수 있음

즉, 임베딩 모델 변경은 운영상 큰 변화입니다.

---

## 6. Vector DB: 벡터를 저장하고 찾는 저장소

Embedding으로 만든 벡터를 저장해두고,
질문 벡터와 가장 비슷한 벡터를 빠르게 찾는 시스템이 Vector DB입니다.

대표적으로는:
- Qdrant
- Weaviate
- Milvus
- Pinecone
- pgvector(PostgreSQL 확장) 등

### 역할
- 벡터 저장
- 유사도 검색
- metadata filter
- hybrid search 지원
- namespace / collection 관리

### metadata 예시
문서 청크와 함께 이런 정보를 저장할 수 있습니다.

```json
{
  "doc_id": "hr_policy_2026_v3",
  "title": "연차 및 휴가 규정",
  "section": "연차 이월",
  "department": "HR",
  "version": 3,
  "updated_at": "2026-03-01"
}
```

이 metadata를 통해:
- 특정 부서 문서만 검색
- 최신 버전만 검색
- 특정 카테고리만 검색

같은 제어가 가능해집니다.

## 7. Top-K: 몇 개까지 가져올 것인가

Top-K는 검색 단계에서 **상위 몇 개를 가져올지** 정하는 값입니다.

예:
- Top-3
- Top-5
- Top-10
- Top-20

### 왜 중요하나
적게 가져오면:
- 진짜 필요한 문서를 놓칠 수 있음

많이 가져오면:
- 관련 없는 문서가 섞임
- LLM 입력이 길어짐
- 비용 증가
- 응답 속도 저하
- 환각 위험 증가

### 실무 감각
- FAQ 성격: Top-3 ~ Top-5
- 복합 질문: Top-8 ~ Top-15
- 후속 Rerank가 있다면 처음엔 더 넓게 Top-20 정도 뽑고 다시 줄이기도 함

즉, Top-K는 **검색 recall과 precision의 균형값**입니다.

---

## 8. Rerank: Top-K를 한 번 더 정밀하게 줄 세우기

처음 검색은 빠르지만 다소 거칠 수 있습니다.
특히 벡터 검색은 “대충 비슷한 것”을 잘 찾지만,
“지금 질문에 진짜 직접 답이 되는 것”까지 정확히 판단하지는 못할 때가 있습니다.

이때 Rerank를 씁니다.

### 흐름
1. Vector DB에서 Top-20 검색
2. Reranker 모델이 질문과 각 문서를 더 정교하게 비교
3. 그중 상위 5개만 LLM에 전달

### 장점
- 노이즈 감소
- 정답률 향상
- LLM 입력 컨텍스트 품질 개선

### 의미
Top-K가 “1차 후보 추출”이라면,
Rerank는 “면접 다시 보고 최종 선발”에 가깝습니다.

---

## 9. LLM 사용: 단순 답변기가 아니라 판단 엔진

AI Agent에서 LLM은 단순히 문장을 자연스럽게 만드는 도구가 아닙니다.

### LLM이 하는 일
- 질문 의도 분류
- 검색 필요 여부 판단
- 검색어 재작성
- retrieved context 해석
- 여러 문서 종합
- 모순 처리
- tool 필요 여부 판단
- 호출할 함수/API 선택
- 응답 생성
- 다음 행동 계획

### 예시 1: 단순 RAG
질문:
- “올해 연차 이월 규정 알려줘”

흐름:
- Retriever가 관련 문서 Top-K 검색
- LLM이 문서를 읽고 답변 생성

### 예시 2: Agent형
질문:
- “내 남은 연차랑 사내 이월 정책 기준으로 올해 안에 써야 할 일수 계산해줘”

흐름:
- 사내 규정 검색
- HR API로 사용자 잔여 연차 조회
- 필요 시 날짜 계산
- LLM이 종합 설명

이 경우 LLM은 검색만 하는 것이 아니라
**어떤 툴을 어떻게 조합할지 판단하는 오케스트레이터** 역할을 합니다.

---

## 10. Tool Calling: 에이전트의 손발

Tool Calling은 LLM이 외부 기능을 호출하는 구조입니다.

### 예시 툴
- SQL 실행
- REST API 호출
- 사내 ERP 조회
- 달력 조회
- 계산기
- 코드 실행기
- 웹 검색
- 메일 요약
- 파일 읽기

### 왜 필요한가
LLM은 학습된 지식만으로는:
- 최신 데이터 조회
- 개인별 데이터 조회
- 정확한 실시간 계산
- 시스템 변경 작업

을 직접 할 수 없습니다.

그래서 도구를 사용해야 합니다.

### 예시
질문:
- “이번 달 매출이 지난달보다 얼마나 줄었어?”

필요한 것:
- DB 조회
- 월별 합계 계산
- 증감률 계산
- 설명 생성

이 흐름에서:
- Tool = DB 조회, 계산
- LLM = 어떤 쿼리를 실행할지 판단하고 결과를 설명

## 11. 재인덱스 제어: 왜 운영에서 중요하나

RAG 시스템은 검색 품질이 핵심인데,
검색 품질은 인덱스 상태에 크게 좌우됩니다.

### 재인덱스가 필요한 상황
- 문서 내용 변경
- 새 문서 추가
- 문서 삭제
- 청크 정책 변경
- 임베딩 모델 변경
- 메타데이터 구조 변경
- 언어 처리 규칙 변경
- 검색 품질 개선 목적

### 재인덱스를 너무 안 하면
- 최신 정보가 반영 안 됨
- 삭제 문서가 검색됨
- 예전 버전이 계속 답변 근거로 나옴

### 재인덱스를 너무 자주 하면
- 비용 증가
- 처리 시간 증가
- 운영 부하 증가
- 서비스 중 검색 일관성 문제 발생 가능

그래서 “재인덱스 제어”가 필요합니다.

---

## 12. 재인덱스 제어의 실무 방식

### 1) 전체 재인덱스
모든 문서를 처음부터 다시 처리

**장점**
- 일관성 높음
- 구조 변경 반영 쉬움

**단점**
- 느림
- 비용 큼
- 대규모 시스템에서는 부담

### 2) 증분 재인덱스
바뀐 문서만 다시 인덱싱

**장점**
- 빠름
- 저렴함
- 운영 효율 좋음

**단점**
- 변경 추적 로직 필요
- 일부 일관성 이슈 가능

### 3) 이벤트 기반 재인덱스
문서 수정 이벤트가 발생하면 자동 처리

예:
- CMS에서 문서 저장
- 이벤트 발생
- 백그라운드 워커가 해당 문서만 재인덱스

### 4) 배치 재인덱스
하루 1회, 1시간마다 등 주기적 실행

### 5) 버전 기반 인덱스
- collection_v1
- collection_v2

처럼 새 인덱스를 만든 후, 검증 완료 시 스위칭하는 방식

이 방식은 서비스 안정성이 좋아 실무에서 자주 사용됩니다.

## 13. 검색 품질을 좌우하는 실제 튜닝 항목

실무에서는 단순히 “LLM 붙였다”가 아니라,
아래 항목들을 계속 조정합니다.

### 검색 전 단계
- chunk size
- chunk overlap
- 문서 정제 방식
- 표/코드/이미지 처리 방식
- 메타데이터 설계

### 검색 단계
- embedding model
- hybrid search 여부
- Top-K
- metadata filtering
- query rewriting
- multi-query retrieval

### 검색 후 단계
- rerank 개수
- LLM에 넣을 문서 수 제한
- context compression
- citation 방식
- answer grounding 규칙

### 운영 단계
- 재인덱스 정책
- 모니터링
- 실패 재처리
- 문서 버전 추적
- 캐시 전략

## 14. Hybrid Search: 키워드 + 의미 검색 같이 쓰기

벡터 검색만으로 충분하지 않은 경우가 있습니다.

예:
- 제품 코드
- 법 조항 번호
- 에러 코드
- 정확한 고유명사
- 숫자/버전 기반 질의

이런 경우 의미 검색만 쓰면 놓칠 수 있어서,
보통 **Hybrid Search**를 사용합니다.

### 구성
- BM25 같은 키워드 검색
- Vector Search
- 결과 병합
- 필요 시 Rerank

즉:
- 키워드는 정확한 문자열 매칭에 강함
- 벡터는 의미 유사도에 강함

둘을 같이 쓰면 더 안정적입니다.

---

## 15. Query Rewriting: 질문을 검색하기 좋게 다시 쓰기

사용자 질문은 검색용으로 적합하지 않을 수 있습니다.

예:
- “그거 휴가 관련해서 남은 거 내년으로 넘기는 거 있잖아 그 기준 뭐였지?”

이 문장을 그대로 검색하면 품질이 떨어질 수 있습니다.

그래서 LLM이 검색용 질문으로 정리합니다.

예:
- “연차 이월 기준 및 미사용 휴가 carry-over 정책”

이것이 Query Rewriting입니다.

### 장점
- 검색 정확도 향상
- 애매한 표현 정리
- 사내 문서 용어와 맞춤

Agent에서는 이 단계가 매우 중요합니다.

---

## 16. Context Window와 왜 Top-K가 연결되는가

LLM은 무한히 많은 문서를 읽을 수 없습니다.
입력 토큰 제한과 비용 문제가 있습니다.

그래서:
- 너무 많은 문서를 넣을 수 없고
- 너무 적게 넣으면 정보 부족

상황이 됩니다.

즉 Top-K는 단순 검색 옵션이 아니라,
**LLM의 context budget 관리 변수**이기도 합니다.

예:
- Top-20을 검색하되
- Rerank 후 Top-5만 전달
- 각 문서는 600 tokens 이하로 제한

이런 식으로 설계합니다.

---

## 17. AI Agent와 RAG의 차이

둘은 겹치지만 완전히 같은 말은 아닙니다.

### RAG
검색된 외부 문서를 LLM에 주입해서 더 정확한 답을 만드는 방식

### AI Agent
LLM이 상황을 판단하고 필요한 검색/도구/행동을 조합하는 구조

즉:

- **RAG는 검색 보강 방식**
- **Agent는 의사결정 및 행동 구조**

실무에서는 둘이 합쳐집니다.

예:
- Agent가 필요 시 RAG 검색 수행
- 검색 결과를 바탕으로 Tool 호출
- 최종 응답 생성

## 21. 실패하는 패턴들

### 1) Chunk가 너무 큼
- 관련 없는 내용까지 같이 검색됨

### 2) Top-K가 너무 큼
- 노이즈가 LLM에 들어감

### 3) 재인덱스 정책 없음
- 오래된 문서가 계속 검색됨

### 4) 임베딩 모델 바뀌었는데 재인덱스 안 함
- 검색 품질 급락 가능

### 5) Tool Calling 검증 없음
- 잘못된 API 호출/과도한 행동 가능

### 6) 출처 관리 없음
- 사용자가 답변을 신뢰하기 어려움

### 7) Rerank 없이 넓게만 검색
- LLM이 정답보다 잡음을 더 많이 읽음

## 23. 아주 쉬운 비유

도서관으로 비유하면:

- **Chunk** = 책을 장/절 단위로 나눠 메모해 둔 것
- **Embedding** = 책 내용의 의미 좌표
- **Vector DB** = 비슷한 의미의 메모를 빨리 찾는 분류 시스템
- **Top-K** = 사서가 관련 메모를 몇 장 꺼내줄지
- **Rerank** = 그중 진짜 필요한 순서로 다시 정렬
- **LLM** = 자료를 읽고 설명하는 전문가
- **Tool Calling** = 전문가가 계산기, 전화, 내부 시스템까지 써서 확인하는 것
- **재인덱스 제어** = 책 내용이 바뀌면 분류표를 다시 업데이트하는 것

---

## 24. 한 번에 정리하는 핵심 요약

### Top-K
검색 결과를 몇 개까지 가져올지 정하는 값  
→ 검색 범위와 노이즈, 비용의 균형

### LLM 사용
검색 결과를 읽고 판단하고 응답하고 도구까지 선택하는 두뇌  
→ 단순 생성기가 아니라 orchestration 엔진

### 재인덱스 제어
지식 저장소를 최신/일관성 있게 유지하는 운영 메커니즘  
→ 정확성과 운영비용의 균형

### Chunk
문서를 검색 가능한 단위로 나누는 전략  
→ 검색 품질의 출발점

### Embedding
텍스트를 의미 벡터로 바꾸는 기술  
→ 의미 검색의 핵심

### Vector DB
벡터를 저장하고 유사도를 기준으로 빠르게 찾는 저장소  
→ RAG 검색 엔진의 핵심 구성요소

### Rerank
후보 검색 결과를 더 정교하게 재정렬  
→ LLM에 들어가는 문맥 품질 개선

### Tool Calling
외부 시스템과 연결해 실제 행동/조회/계산을 수행  
→ Agent를 단순 챗봇에서 업무 수행자로 확장

---

## 25. 실무 한 줄 결론

AI Agent 시스템에서:

- **Chunk / Embedding / Vector DB** 는 “기억을 저장하는 방식”
- **Top-K / Rerank** 는 “어떤 기억을 꺼낼지 고르는 방식”
- **LLM 사용** 은 “꺼낸 기억과 현재 상황을 이해하고 판단하는 방식”
- **Tool Calling** 은 “판단 결과를 행동으로 옮기는 방식”
- **재인덱스 제어** 는 “기억 저장소를 계속 최신 상태로 유지하는 방식”

즉, 좋은 Agent는 LLM 하나가 아니라  
**검색 품질 + 컨텍스트 품질 + 도구 연결 + 운영 품질**이 함께 좋아야 합니다.

## 28. 마무리

AI Agent를 이해할 때 많은 분들이 먼저 LLM만 떠올리지만,
실제로 정확도와 실용성을 좌우하는 것은 그 앞뒤의 시스템입니다.

특히 실무에서는 아래 세 가지가 매우 중요합니다.

- 얼마나 잘 찾는가
- 찾은 것을 얼마나 잘 해석하는가
- 최신 정보를 얼마나 안정적으로 유지하는가

이 관점에서 보면:

- Top-K는 **얼마나 잘 찾는가**
- LLM 사용은 **얼마나 잘 해석하는가**
- 재인덱스 제어는 **얼마나 최신성을 유지하는가**

를 담당한다고 보면 이해가 쉽습니다.
